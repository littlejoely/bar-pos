### 0.2 术语约定

| 术语 | 定义 |
|---|---|
| 开台 | 将空桌变为营业中桌台，并记录人数、开台时间和备注 |
| 下单 | 确认首轮菜品，使订单从待下单进入已下单状态 |
| 加菜 | 已下单后创建一轮尚未提交的新菜品记录 |
| 本轮加菜 | 自点击“加菜”开始到点击“下单”前的临时菜品集合 |
| 退菜 | 对已下单菜品做软删除，保留原记录、退菜原因和审计信息 |
| 应收 | 应用菜品优惠、整单优惠、优惠券和抹零后的订单金额 |
| 实收 | 各支付流水与优惠券实际收入之和，扣除退款后为净实收 |
| 结清 | 已收金额达到应收金额，允许执行清台 |
| 清台 | 结束桌台占用并将桌台恢复为空桌；订单历史继续保留 |
| 制作单 | 下单、加菜或补打时生成的出品任务单 |
| 划菜 | 将制作单中的某条菜品标记为已完成 |
| 售卖状态 | 商品“在售”或“已下架”；已下架商品不出现在点单页 |

## 1. 产品概述

### 1.1 产品定位

Silver Lining POS 是面向单门店酒吧及轻餐饮场景的 Web 收银与出品协同系统。它强调低部署成本、桌台业务连续性、清晰的优惠与支付明细、可追溯操作记录，以及在无专用客户端情况下通过浏览器完成主要营业流程。

### 1.2 目标用户

| 用户角色 | 核心诉求 | 当前权限模型 |
|---|---|---|
| 收银员 | 开台、点单、结账、清台、查订单 | 已按角色权限控制，默认不含退款及优惠权限 |
| 服务员 | 开台、点单、加菜、备注、查看制作进度 | 已按角色权限控制，默认无收款和经营数据权限 |
| 出品人员 | 查看制作单、划菜、完成制作单 | 已按角色权限控制，默认进入制作单记录页 |
| 店长/管理员 | 经营管理、基础资料与人员管理 | 已支持店长和超级管理员；超级管理员信息对其他角色隔离 |

### 1.3 产品目标

1. 完整支撑“开台 → 点单 → 下单 → 加菜 → 优惠 → 收款 → 结清 → 清台”闭环。
2. 保证已下单、已支付和已退款信息可追溯，不通过物理删除掩盖业务事实。
3. 将前台订单与制作单联动，减少漏单、重复出品和沟通成本。
4. 为店长提供订单、支付、优惠、退款和制作单的历史查询及 Excel 导出。
5. 在单机或单实例服务器上保持简单、稳定、可备份的部署方式。

### 1.4 非目标与当前边界

- 不对接微信、支付宝官方支付网关；扫码支付为人工确认支付渠道。
- 不提供会员、库存、采购、排班、交接班和财务总账；敏感操作临时授权仍属于后续阶段。
- 不直接控制实体打印机；当前打印依赖浏览器打印或生成制作单记录。
- 不支持多门店、多租户或多后端实例并发写入。
- 邮件导出代码已预留，但当前业务决定暂不配置 SMTP 服务。
- 当前订单历史由前端加载全量订单后筛选，尚未实现后端分页与短时缓存。
- 当前已完成登录、RBAC、CSRF 和基础审计，但业务 JSON 存储、单机限流与部署安全仍需正式验收；未配置 HTTPS 时不得直接暴露于不可信公网。

### 1.5 当前数据快照

以下数据仅描述 2026-07-13 的运行基线，不作为固定上限：

## 2. 信息架构与全局体验

### 2.1 顶级导航

顶栏导航在整个页面水平居中，包含：

1. **桌台**：桌台状态、经营概览、制作单面板；
2. **订单历史**：经营统计、订单列表、订单详情、打印、退款和导出；
3. **制作单记录**：仅在制作单模式开启时显示；
4. **设置**：类别、商品、区域、桌台、优惠券和制作单管理。

Tab 切换使用独立滑动高亮层，按实际 Tab 宽度平滑过渡。进入某桌台订单详情时，“桌台”仍保持选中态。

### 2.2 全局视觉与交互原则

- 主品牌色用于主操作、选中态和关键高亮；成功状态使用绿色，警告与超时使用红色或橙色。
- 金额统一使用人民币符号并固定保留两位小数，例如 `¥320.00`。
- 需要微调的数字输入统一使用输入框内部“减号 / 加号”步进控件。
- 表格表头固定，数据在表格内部滚动，分页栏固定于底部。
- 表格记录总数位于分页控件左侧，与页码区域间距 20px。
- 表格每页条数支持 10、20、50、100，切换条数时分页控件位置保持稳定。
- 页面切换、列表选中、制作单划菜和导航切换避免闪烁或中间态残影。

### 2.3 页面加载原则

- 进入订单详情前应先确认目标桌台和订单状态，避免先渲染错误页面再闪烁切换。
- 进入订单历史或制作单记录时请求后端最新数据；当前不使用跨页面缓存。
- 订单加菜的临时数据由后端订单记录保存，离开页面后可恢复，直到提交或明确取消。

## 3. 核心业务对象与状态

### 3.1 桌台状态

| 后端状态/展示状态 | 中文标签 | 允许操作 |
|---|---|---|
| `empty` | 空桌 | 开台、作为转台目标 |
| `occupied` + 无订单或待下单 | 待下单 | 点单、修改人数、取消订单 |
| `occupied` + 已下单未结清 | 待结账 | 加菜、结账、转台、并台 |
| 已下单且余额为 0 | 已结清 | 清台、撤销部分收款后继续调整 |
| `pending_cleanup` | 已结清 | 清台 |

桌台是否“已结清”不只依赖桌台字段，而是结合订单应收与付款汇总计算。

### 3.2 订单状态

| 状态 | 含义 | 进入条件 | 退出条件 |
|---|---|---|---|
| `pending` | 待下单 | 开台后首次创建订单 | 点击下单或取消订单 |
| `submitted` | 已下单 | 首轮下单成功 | 结账、取消、并台等 |
| `served` | 已上菜/兼容状态 | 历史兼容 | 结账、取消、并台 |
| `paid` | 已支付 | 收款达到应收并确认结账 | 清台不改变订单支付状态 |
| `canceled` | 已取消 | 撤销活动订单 | 终态 |
| `merged` | 已并台 | 源订单合并至目标订单 | 终态 |

### 3.3 订单菜品状态

- 普通菜品：属于已确认订单；
- `addition_pending=true`：本轮加菜临时记录；
- `returned=true`：已退菜，保留名称、数量、原价、原因和时间；
- 菜品可附加备注、折扣、减免、赠菜数量及对应原因；
- 同一商品在不同加菜轮次必须形成独立订单行，不得合并到上一轮已下单菜品。

### 3.4 制作单状态

- 制作中：`archived=false`；
- 菜品完成：对应明细 `completed=true` 并记录 `completed_at`；
- 已完成：全选或点击完成后制作单归档，记录 `archived_at`；
- 制作时长从制作单创建时间起动态计算，超过 10 分钟显示红色警告。

## 4. 功能需求

### 4.1 桌台页面

#### 4.1.1 经营概览

桌台页右上角展示：

- 待结账桌数、人数和金额；
- 今日已支付订单数、人数和营业总额；
- 所有金额固定两位小数。

#### 4.1.2 区域筛选与桌台卡片

- 按区域筛选桌台，默认可查看全部；
- 卡片显示桌号、人数、已确认菜品数、已下单总额和状态；
- 尚未提交的本轮加菜不得计入桌台卡片总额和菜品数量；
- 已结清但未清台的桌台使用绿色卡片和“已结清”标签；
- 点击不同状态桌台进入对应操作：空桌开台、活动桌点单或结账、已结清桌清台。

#### 4.1.3 开台

- 弹窗显示桌号、默认人数、数字键盘、桌台备注；
- 默认人数在弹窗打开时立即高亮，首次输入直接替换；
- 人数范围 1–50；
- 开台后记录 `opened_at`，桌台进入占用状态；
- 桌台默认人数由设置页维护。

#### 4.1.4 转台与并台

- 转台目标必须为空桌；源桌订单、人数和开台时间迁移到目标桌；
- 并台目标必须为已开台且存在活动订单；
- 并台后源订单标记为 `merged`，源桌释放；
- 操作必须写入源订单和目标订单操作日志。

#### 4.1.5 清台

- 仅结清订单允许清台；
- 清台记录订单 `cleared_at` 和操作日志；
- 桌台恢复为空桌，人数、开台时间和订单引用清空。

### 4.2 点单与订单明细

#### 4.2.1 分类导航与商品卡片

- 分类导航位于顶部卡片，Tab 整体水平居中；
- 分类较多时支持横向滚动，但不展示滚动条；
- 商品卡片显示中文名、英文名、售价、单位或 ABV；
- 商品售价固定两位小数；
- 已下架商品和没有在售商品的空分类不出现在点单页。

#### 4.2.2 首轮点单

- 新加入菜品显示在菜品明细顶部；
- 首轮同款菜品可累加数量；
- 清空按钮清除当前尚未下单菜品；
- 点击“下单”后订单进入 `submitted`；
- 制作单模式开启时生成“下单”制作单。

#### 4.2.3 加菜

- 已下单订单点击“加菜”进入加菜模式；
- 已下单菜品仅文字置灰，不使用灰色填充底框；
- 已下单菜品在加菜模式下不可修改；
- 本轮新增菜品排在顶部，默认选中新入列第一条；
- 同款商品只允许在本轮加菜内部合并，禁止合并到历史订单行；
- 加菜模式的清空只清除本轮加菜；
- 离开页面不自动清除本轮加菜，重新进入可恢复；
- 点击“取消加菜”才彻底删除缓存；
- 只有点击“下单”才确认加菜并生成制作单。

#### 4.2.4 菜品与整单备注

- 每条菜品可设置或清空备注；
- 订单可设置整单备注；
- 整单备注在菜品明细列表上方以独立卡片展示；
- 制作单同时展示整单备注和菜品备注。

#### 4.2.5 菜品操作

未结清且金额规则允许时，支持：

- 修改数量；
- 菜品折扣；
- 菜品金额减免；
- 赠菜及赠菜数量；
- 退菜；
- 每次操作记录原因或备注，并进入操作日志。

#### 4.2.6 退菜

- 已下单菜品退菜不能从明细中物理删除；
- 菜品名称后显示“退”标签；
- 价格划线、整行文字置灰；
- 退菜理由选项：多点、错点、顾客要求、其他；
- 选择“其他”时可填写补充说明；
- 退菜弹窗内容整体居中对齐。

#### 4.2.7 补打制作单

- “补打小票”统一命名为“补打制作单”；
- 进入补打模式后，菜品明细变为复选列表；
- 支持全选、确认和按菜品数量选择补打份数；
- 已退菜和未提交加菜不可补打；
- 制作单模式关闭时入口隐藏。

### 4.3 优惠与应收计算

#### 4.3.1 菜品级优惠

- 赠菜：按赠送数量扣除对应金额；
- 折扣：保存折扣减免比例，页面显示实际折数；
- 减免：保存固定减免金额，不能超过该菜品可减金额；
- 退菜：退菜行金额计为 0。

#### 4.3.2 整单优惠

- 整单折扣：范围 0–100；
- 整单减免：不能超过折后订单金额；
- 抹零：默认取应收金额个位尾数，支持修改；
- 免单：执行前提示“免单会取消所有优惠”，确认后清除原优惠并将订单免单；再次点击需确认撤销免单。

#### 4.3.3 优惠券

- 结账页只展示设置中已存在的优惠券卡片；
- 可选择每种优惠券的使用数量，单种最多 99 张；
- 优惠券可撤销，撤销后可重新选择；
- 优惠券面值计入抵扣，应付收入按售价计入实收，面值与售价差额计入优惠；
- 例如“90 代 100”：付款信息记录收入 ¥90.00，优惠汇总记录优惠 ¥10.00。

#### 4.3.4 金额公式

```text
菜品行金额 = max(0,
  单价 × (数量 - 赠菜数量)
  × (1 - 折扣减免百分比)
  - 菜品固定减免
)

菜品小计 = Σ 未退菜菜品行金额
整单折后金额 = 菜品小计 × (1 - 整单折扣减免百分比)
应收金额 = max(0,
  整单折后金额
  - 整单减免
  - 优惠券面值
  - 抹零
)

优惠券实际收入 = Σ 优惠券售价 × 使用数量
现金类实收 = Σ 支付流水金额
已收总额 = 现金类实收 + 优惠券实际收入
还差金额 = max(0, 应收金额 - 已收总额)
```

#### 4.3.5 结清后的金额保护

- 当已收金额大于 0 且还差金额为 0 时，禁止继续退菜、赠菜、折扣、减免、整单优惠和抹零；
- 必须先撤销一笔或部分收款，使订单重新出现未收余额；
- 新增优惠不得超过当前未收余额允许的调整范围；
- 优惠券撤销不受该锁限制，撤销后可重新计算并选择其他操作。

### 4.4 支付与结账

#### 4.4.1 支付入口

结账页“选择支付方式”仅保留：

1. 扫码支付；
2. 现金；
3. 优惠券抵扣。

#### 4.4.2 扫码支付

- 点击后弹出“扫码支付”弹窗；
- 金额区域格式为 `¥ 320.00    修改金额`；
- 下方提供微信、支付宝两个选项；
- 点击渠道后登记对应收款流水，不调用真实支付接口；
- 右侧付款明细显示“微信/支付宝 收款 ¥xx.xx”。

#### 4.4.3 现金支付

- 现金流程与扫码支付金额确认逻辑一致；
- 确认后登记现金收款流水。

#### 4.4.4 修改金额键盘

四列四行等宽等高：

```text
1  2  3  退格
4  5  6  清空
7  8  9  确认（占两行）
.  0  00
```

- 退格使用向左五边形图标；
- 弹窗打开时只高亮金额文字，不高亮输入框；
- 高亮必须同步出现，无延迟；
- 首次输入直接替换默认金额。

#### 4.4.5 组合支付与撤销

- 一个订单可登记多笔不同支付方式流水；
- 多种渠道共同支付时，订单付款方式为“组合支付”；
- 每笔收款均可撤销；
- 撤销后应收、实收、还差金额和桌台状态同步更新；
- 还差金额未归零时清台按钮为灰色禁用，归零后变绿并允许清台。

#### 4.4.6 完成结账

- 订单必须包含菜品；
- 已收总额不得小于应收金额；
- 完成后订单状态改为 `paid`，记录 `paid_at`、付款方式和实收；
- 桌台进入已结清/待清台状态；
- 多收部分视为溢收，当前不自动找零或退款。

### 4.5 制作单

#### 4.5.1 功能开关

- 设置页提供“制作单模式”开关；
- 关闭后隐藏桌台页制作单面板、补打制作单、顶栏制作单记录及相关生成能力；
- 加菜确认在关闭模式下仍须完成订单提交，但不生成制作单。

#### 4.5.2 桌台页制作单面板

- 原“待出品小票”统一命名为“制作单”；
- 无制作单时面板折叠在页面底部，不遮挡桌台；
- 每张制作单卡片尺寸固定，常驻展示最多 7 条菜品；
- 标题显示“桌台号 · 制作时长”和下单时间；
- 10 分钟内制作时长为黑色，超过 10 分钟为红色；
- 菜品之间不使用分隔横线；
- 显示菜品备注和订单备注。

#### 4.5.3 划菜交互

- 菜品使用自定义 SVG 动画复选框；
- 勾选色在窗口失焦时保持一致，不变成浏览器黑色默认勾；
- 已划菜品文字置灰并整行划线；
- 已划菜品平滑移动至底部，未出品菜品自动上移；
- 动画包含短暂停顿、缓慢位移和稳定颜色，避免瞬移和闪烁；
- “全选”“完成”是制作单卡片内部底部悬浮按钮，可轻微遮挡数量信息。

#### 4.5.4 制作单记录

- 页面结构参考订单历史，但不展示经营统计卡片；
- 默认今日，支持今日、昨日、近 7 天、近 30 天；
- 支持状态、类型和关键词筛选；
- 表头固定、列表内部滚动、固定分页；
- 点击记录从右侧打开详情抽屉；
- 详情展示制作单号、订单号、复制按钮、桌台、类型、状态、制作时长、下单/完成时间、备注和单行菜品明细；
- 导出遵循全部当前筛选条件，跨分页导出 Excel，包含制作单汇总和菜品明细。

### 4.6 订单历史与经营分析

#### 4.6.1 日期与筛选

- 默认展示今日；
- 日期面板提供今日、昨日、近 7 天、近 30 天；
- 月份使用中文；
- 支持状态、付款方式、订单号和桌台号筛选；
- 进入页面时重新请求最新订单，不使用跨页面缓存。

#### 4.6.2 指标卡片

1. 营业额；
2. 订单数；
3. 客单价；
4. 优惠金额；
5. 付款方式：微信、支付宝、现金的金额和占比。

金额与百分比使用加粗浅灰竖线 `｜` 分隔。

#### 4.6.3 同比与环比口径

- 当前周期由用户选择的日期范围确定；
- 环比周期为当前周期之前、长度完全相同的连续时间范围；
- 同比周期为当前日期范围整体向前移动一年；
- 营业额以已支付订单净实收为基础，扣除退款；
- 客单价 = 营业额 / 已支付订单数；
- 支付渠道金额按退款后的净额比例分摊。

#### 4.6.4 订单列表

- 字段：序号、订单号、桌台、状态、应收、实收、付款方式、下单时间、结束时间；
- 下单时间以首轮下单为准；
- 结束时间优先使用清台时间，其次取消或并台时间；
- 表头固定，列表内部滚动；
- 点击订单进入右侧订单详情抽屉。

#### 4.6.5 订单详情

- 返回按钮为无边框左箭头；
- 顶部第一行：桌台、人数、状态；
- 顶部第二行：开台时间、最后付款时间、清台时间；
- 菜品明细沿用点单页视觉，增加每条菜品首轮或加菜下单时间；
- 标签展示赠菜、折扣、减免、退菜等状态；
- 优惠汇总包含赠菜、折扣菜品、减免菜品、抹零和优惠券优惠；
- 付款信息展示应收、实收、付款方式及每笔支付时间；
- 优惠券同时出现在优惠汇总和付款收入明细；
- 底部展示细颗粒度操作日志。

#### 4.6.6 打印订单详情

- 订单详情右上角提供打印按钮；
- 打印内容包含店名、订单结构、打印时间和详情模块；
- 打印样式适配 A4，避免顶部溢出、比例过大和跨两页；
- 打印弹窗按钮必须可用，不得处于错误禁用状态。

#### 4.6.7 退款

- 仅已支付订单允许退款；
- 默认金额为当前可退金额，可打开修改金额键盘；
- 必填退款原因；
- 支持多次部分退款，累计不得超过净实收；
- 记录退款流水、时间、原因和部分/全额退款状态；
- 当前退款为系统登记，不调用支付渠道原路退款接口。

#### 4.6.8 导出

- 本地导出按当前筛选结果导出全部匹配订单，不受分页限制；
- Excel 包含订单汇总、菜品明细和付款明细；
- 邮件导出仅在 SMTP 配置完整时可用；当前业务默认不配置。

### 4.8 登录、用户与权限（第一阶段已实现）

本模块的认证、用户、角色、权限、会话和基础审计能力已在 v5.0 落地，系统已从“共享终端、匿名操作”升级为“实名用户、角色授权、关键操作可追溯”。敏感操作的单次临时授权尚未实现，继续作为下一阶段 P0 能力。

#### 4.8.1 设计原则

1. **默认拒绝**：未明确授予的权限一律不可用；
2. **前后端双重校验**：前端隐藏或禁用入口，后端 API 必须再次鉴权；
3. **最小权限**：员工仅获得完成岗位职责所需权限；
4. **实名审计**：所有关键业务记录真实操作人，不再统一写“收银员”；
5. **敏感操作可授权**：无权限员工可请求店长或超级管理员进行单次临时授权；
6. **业务连续性**：退出登录、会话过期或账户停用不得破坏进行中的订单数据；
7. **凭据安全**：登录密码和短密码只保存强哈希，不保存、记录或返回明文。

#### 4.8.2 登录方式

系统同时支持两种登录方式：

1. **账号 + 密码**：用于超级管理员、店长、后台管理和首次登录；
2. **短账号 + 短密码**：用于收银台或触摸屏快速登录和切换员工；短账号为 2 至 12 位字母或数字，短密码为 4 至 8 位数字。

业务规则：

- 不开放员工自主注册，账户只能由有权限的管理员创建；
- 首次部署必须初始化唯一超级管理员；
- 超级管理员初始化时必须同时设置个人短账号和短密码；
- 新用户首次密码登录必须修改密码；
- 短密码与登录密码独立设置、独立哈希，不能互相推导；
- 忘记登录密码或短密码由管理员重置，不提供公网“找回密码”邮件流程；
- 已停用、锁定或已删除凭据的用户不得登录；
- 登录成功后进入其有权访问的默认页面；没有桌台查看权限的用户不得进入桌台页。

#### 4.8.3 会话管理

- 使用服务端会话和 `HttpOnly` Cookie，不将认证 Token 放入 `localStorage`；
- Cookie 在生产环境启用 `Secure`、`HttpOnly` 和合适的 `SameSite` 策略；
- 默认闲置 30 分钟后锁屏，最长会话 12 小时；两个时长应可配置；
- 锁屏后可用短账号 + 短密码快速恢复或切换员工；
- 主动退出、账户停用、密码重置后立即使对应会话失效；
- 同一账户允许的并发会话数量应可配置，默认允许 2 个；
- 页面刷新可恢复有效会话，但不能恢复已失效权限；
- 会话中保存用户 ID、当前角色集合、登录方式、创建/最后活动时间和终端信息。

#### 4.8.4 登录风控

- 同一账号或员工号连续失败 5 次，临时锁定 15 分钟；
- 成功登录后清除连续失败计数；
- 管理员可手动解锁；
- 登录失败提示不区分“账号不存在”和“密码错误”，避免用户枚举；
- 记录成功/失败、IP、User-Agent、时间和失败原因代码；
- 密码最少 8 位，必须同时包含字母和数字；超级管理员建议至少 12 位；
- 禁止使用当前密码作为新密码，后续可扩展最近密码历史。

#### 4.8.5 用户管理

设置页新增“用户管理”，功能包括：

- 列表字段：序号、姓名、当前账号标识、员工号、登录账号、短账号、手机号、角色、状态、最后登录时间和操作；
- “当前账号”使用紧凑标签，不得挤压员工号等字段；
- 登录密码和短密码不在列表展示，后端也不返回明文或可还原内容；
- 新增用户：姓名、员工号、登录账号、短账号、初始角色、手机号（可选），同时设置初始登录密码和短密码；
- 编辑用户姓名、员工号、登录账号、短账号、联系方式和角色；
- “重置凭证”必须先选择重置账号密码或短密码，每次只能重置一种；重置登录密码后强制下次登录修改；
- 停用、启用、临时锁定和解锁用户；
- 查看账户状态、角色、创建时间、最后登录时间和最后操作时间；
- 查看并终止用户当前会话；
- 搜索姓名、账号或员工号，按角色和状态筛选；
- 未产生成功业务记录、不是当前账号且不是超级管理员的用户允许删除；已产生业务记录的用户只能停用，接口需给出可理解原因；
- 超级管理员账号不能被其他角色查看、编辑、删除、停用、分配或重置凭证；超级管理员凭证只能由本人重置；
- 当前唯一超级管理员不得被删除、停用或降权，系统至少保留一个有效超级管理员；
- 用户管理工具栏除“添加用户”外，搜索和筛选全部靠左，添加按钮靠右；
- 删除等操作按钮必须常驻可见，不能藏在横向滚动区域。

用户状态：

| 状态 | 含义 | 是否允许登录 |
|---|---|---|
| `active` | 正常 | 是 |
| `password_change_required` | 待修改密码 | 仅允许进入改密流程 |
| `locked` | 登录失败临时锁定或管理员锁定 | 否 |
| `disabled` | 已停用 | 否 |

##### 4.8.5.1 作品集访客

- 系统可通过环境变量启用公开访客账号，登录页以浅灰小字展示访客账号与登录密码；
- 访客可以查看除超级管理员以外的页面、数据、组件和按钮；
- 用户、角色及各类管理页面保留完整组件和操作按钮；访客对非本人创建记录的按钮置灰，并提示“访客用户不可操作其他用户创建的数据”；
- 商品、类别、区域、桌台、优惠券、普通用户、普通角色和订单等可归属对象允许访客创建；
- 访客只能继续编辑、排序、批量处理或删除本人创建的数据，系统既有数据及其他用户创建的数据只读；
- 访客创建的新用户强制继承 `own_created` 数据范围，不能通过分配其他角色绕过数据边界；
- 制作单全局开关等无法归属于单条记录的系统级设置保持只读；
- 数据归属由后端按创建人 ID 强制校验，前端禁用只用于解释交互；
- 正式非演示环境可关闭公开访客账号或替换演示凭据。

#### 4.8.6 RBAC 权限模型

权限关系采用：`用户 → 角色 → 权限`。一个用户可分配多个角色，最终权限取并集；显式禁止权限作为后续扩展，不纳入第一版。

系统首次初始化会创建五类默认岗位角色：

| 角色 | 定位 | 默认能力 |
|---|---|---|
| 超级管理员 | 系统所有者 | 全部业务、系统、用户、角色和安全配置 |
| 店长 | 门店经营负责人 | 全部营业操作、经营数据、退款、敏感授权和基础资料管理，不可修改超级管理员 |
| 收银员 | 收款与桌台运营 | 开台、点单、收款、结账、清台、普通订单查询；敏感金额操作需授权 |
| 服务员 | 前场服务 | 查看桌台、开台、点单、加菜、备注、提交退菜请求；无收款和经营数据权限 |
| 出品人员 | 后场出品 | 查看制作单、划菜、完成制作单和制作单历史；无订单金额与设置权限 |

界面不展示“系统预设”标签。超级管理员角色不可删除、改名、改代码或减少权限；其他角色无论是否由初始化过程创建，都与普通角色完全一致，可编辑名称、角色代码、说明、默认首页和权限，也可删除。删除使用中的角色时必须选择接替角色，后端先完成全部用户改派再删除原角色；接替角色不得是被删除角色或超级管理员。

#### 4.8.7 权限颗粒度

权限至少拆分为以下资源和动作：

| 模块 | 权限代码示例 | 权限动作 |
|---|---|---|
| 桌台 | `table.view/open/edit_guests/transfer/merge/clear` | 查看、开台、改人数、转台、并台、清台 |
| 点单 | `order.create/add_item/change_quantity/addition/submit/cancel` | 创建、加菜、改数量、确认加菜、下单、取消订单 |
| 菜品调整 | `order.return/gift/discount/reduction/free/round_down` | 退菜、赠菜、折扣、减免、免单、抹零 |
| 支付 | `payment.collect/revoke/checkout/refund` | 收款、撤销收款、完成结账、退款 |
| 制作单 | `ticket.view/check/complete/reprint/history` | 查看、划菜、完成、补打、历史 |
| 订单历史 | `history.view/detail/print/export` | 查看、详情、打印、导出 |
| 商品 | `menu.view/create/edit/delete/batch/export` | 查看、新增、编辑、删除、批量、导出 |
| 桌台设置 | `table_config.view/edit` | 查看和维护区域、桌台 |
| 优惠券 | `voucher.view/create/edit/delete` | 查看、新增、编辑、删除 |
| 系统设置 | `system.production_ticket` | 开关制作单模式 |

#### 4.8.9 敏感操作临时授权（下一阶段 P0）

以下操作默认视为敏感操作：

- 退款；
- 免单；
- 撤销收款；
- 对已下单菜品执行退菜、折扣、减免或赠菜；
- 取消已下单订单；
- 删除商品、类别、桌台、区域或优惠券；
- 导出经营数据；
- 用户、角色与权限变更。

授权流程：

1. 无权限用户点击操作，系统不直接执行；
2. 弹出“需要管理授权”弹窗，展示操作、订单、金额影响和申请人；
3. 店长或超级管理员输入本人账号密码，或短账号 + 短密码；
4. 后端校验授权人是否有效且具备该权限；
5. 授权只对本次请求、当前业务对象和短时间窗口有效，不提升申请人的持续权限；
6. 请求执行成功或失败后授权凭证立即失效；
7. 审计日志同时记录申请人、授权人、原因、业务对象、前后金额和结果。

退款、免单等金额操作仍必须填写原有业务原因，管理员授权不能替代业务原因。

#### 4.8.10 权限在前端的表现

- 无模块查看权限：不显示顶栏或设置 Tab；
- 有查看、无操作权限：展示数据但隐藏或禁用操作按钮；
- 可临时授权：按钮保留，点击后进入授权流程；
- 禁用按钮必须提供原因提示，不能只改变颜色；
- 页面加载后权限发生变化时，应重新拉取权限或强制会话刷新；
- 直接输入无权限页面 URL 时跳转至 403 页面或用户默认首页；
- 出品人员默认登录后进入制作单区域。

#### 4.8.11 后端鉴权要求

- 所有非健康检查和登录接口默认要求有效会话；
- 每个写接口声明所需权限，不能依赖前端传入角色；
- 查询接口按权限过滤敏感字段，例如服务员不应获取完整经营统计；
- 高风险接口支持单次授权上下文，并验证授权业务对象与请求一致；
- 用户、角色、权限或账户状态变化后，应使权限缓存和相关会话及时失效；
- 401 表示未登录或会话失效，403 表示已登录但权限不足；
- 鉴权失败不得修改任何业务数据。

#### 4.8.12 审计日志

审计范围包括：

- 登录、退出、锁屏、解锁、失败登录；
- 用户、角色、权限和会话变更；
- 开台、下单、加菜、退菜、转台、并台、清台；
- 折扣、减免、赠菜、免单、优惠券和抹零；
- 收款、撤销收款、结账和退款；
- 商品、桌台、区域、优惠券和系统设置变更；
- 打印、补打和数据导出；
- 敏感操作申请、授权、拒绝和执行结果。

审计日志字段至少包括：日志 ID、时间、用户 ID、用户姓名、角色快照、终端/IP、模块、动作、业务对象、变更前、变更后、业务原因、授权人、执行结果和错误代码。普通用户不得编辑或删除审计日志。

#### 4.8.13 权限模块数据存储

登录密码、短密码、角色、权限、会话和审计日志不得存放在普通 JSON 文件中。当前已使用 SQLite；若系统准备多终端或长期正式运营，应迁移至 PostgreSQL。

推荐技术基线：

- SQLAlchemy：数据访问；
- Alembic：数据库结构迁移；
- Argon2id（优先）或 bcrypt：登录密码与短密码哈希；
- 服务端会话存储：数据库或 Redis（规模扩大后）；
- 数据库级唯一约束：登录账号、员工号和角色编码。

## 7. 非功能需求

### 7.1 性能

- 局域网或正常互联网环境下首屏目标 ≤ 3 秒；
- 常用点单操作反馈目标 ≤ 300ms（不含网络故障）；
- 列表滚动、选中和制作单动画保持稳定，不出现明显掉帧；
- 当前全量订单接口适用于小规模单店，订单数达到万级前应改造后端筛选与分页。

### 7.2 数据一致性

- 金额后端按两位小数归一化，前端显示固定两位；
- 订单、桌台和制作单关键变更应在一次请求中完成；
- JSON 写入优先使用临时文件 + `os.replace` 原子替换；
- 菜单导入、名称迁移前必须创建备份；
- 不允许同时运行多个可写 Flask 实例。

### 7.3 可用性

- 关键操作有明确确认弹窗：免单、撤销免单、退菜、清台、退款等；
- 禁用状态必须同时体现视觉和点击行为；
- 输入默认值需要直接选中，减少“先清空再输入”；
- 空状态不应占据不必要空间或遮挡核心桌台区域。

### 7.4 兼容性

- 以 Chromium 内核桌面浏览器为主要目标；
- 支持鼠标、触控板和触摸屏；
- 打印目标为 A4 浏览器打印；
- 小尺寸窗口提供基础响应式布局，但主操作场景仍以桌面宽屏为准。

### 7.5 安全与权限

当前已实现：

- 登录密码和短密码使用 Argon2id 强哈希，仅保存不可逆摘要；
- 服务端会话使用随机令牌摘要，浏览器 Cookie 为 HttpOnly，并支持生产环境 Secure；
- 所有状态变更 API 校验 CSRF Token，业务接口默认要求有效会话；
- 写接口声明细粒度权限，前端入口按相同权限隐藏或禁用；
- 登录失败锁定、IP + 账号速率限制、闲置锁屏、会话数量限制均支持环境变量配置；
- 账户停用和凭据重置立即撤销会话，角色变更在后续请求实时生效；
- 登录、用户、角色、会话及业务写操作写入审计日志，订单明细记录真实操作人；
- 超级管理员用户、角色和其审计信息对非超级管理员隐藏；
- CORS 来源受环境变量限制，并返回基础浏览器安全响应头。

正式公网运营前仍必须完成：

- 配置 HTTPS、`POS_COOKIE_SECURE=true`、可信反向代理和严格 CORS 来源；
- 将进程内登录限流升级为 Redis 或网关共享限流，以支持多 Worker；
- 对业务 JSON 文件实施单写进程约束、自动备份和恢复演练，或迁移至 PostgreSQL；
- 实现敏感操作单次临时授权及授权人审计；
- 完成渗透测试、依赖漏洞扫描、日志轮转和监控告警。

### 7.6 可维护性

- 前端 TypeScript 构建必须通过；
- 前后端依赖必须使用锁文件；
- 业务字段变更同步更新 PRD、接口、数据兼容和导出；
- 不允许依赖仅存在于开发机而未写入依赖文件。

## 8. 验收标准

### 8.1 主流程验收

1. 空桌可开台并使用默认人数；
2. 可连续添加菜品且页面不跳回桌台；
3. 首轮下单后桌台金额仅包含已提交菜品；
4. 加菜离开页面后可恢复，取消加菜后彻底清除；
5. 提交加菜后生成独立菜品行和制作单；
6. 可应用菜品优惠、整单优惠、优惠券和抹零；
7. 可使用微信、支付宝、现金及优惠券组合结算；
8. 未结清不能清台，结清后桌台变绿；
9. 清台后桌台恢复为空桌，订单历史保留完整时间和明细。

### 8.2 金额验收

- 左右应收金额一致；
- 优惠存在时显示灰色删除线原价；
- 所有金额固定两位小数；
- 优惠券收入和优惠差额口径正确；
- 撤销支付、撤销优惠券和退款后金额同步；
- 已结清状态禁止继续造成应收下降的优惠操作。

### 8.3 制作单验收

- 制作单开关可完整隐藏或恢复入口；
- 下单、加菜、补打能生成正确明细；
- 划菜后动画平滑、状态不闪烁、已完成菜品移动到底部；
- 超过 10 分钟红色预警；
- 制作单详情和 Excel 导出数据一致。

### 8.4 管理与历史验收

- 商品筛选、分页、排序、售卖状态和批量修改正确；
- 全选覆盖所有筛选结果而非仅当前页；
- 商品、订单、制作单导出遵循筛选条件；
- 订单详情展示支付、优惠券、退款和操作日志；
- A4 打印预览不溢出且主要结构保持完整。

### 8.5 登录与权限验收

#### 登录与会话

1. 未登录访问任意受保护页面或 API 时被拒绝并进入登录页；
2. 账号密码和短账号 + 短密码均可正常登录；
3. 首次登录强制修改密码，不能绕过进入业务页面；
4. 连续失败 5 次后锁定 15 分钟，管理员可解锁；
5. 闲置超时进入锁屏，短密码解锁后返回原业务页面且订单数据不丢失；
6. 退出、停用用户或重置密码后旧会话立即失效；
7. Cookie 中不存在可被 JavaScript 读取的明文凭据或长期 Token。

#### 用户与角色

1. 超级管理员可创建、编辑、停用用户并分配角色；
2. 系统始终至少保留一个有效超级管理员；
3. 除超级管理员本人外，其他角色看不到超级管理员用户与角色，也不能通过直接 API 操作；
4. 五类默认岗位角色的初始权限符合矩阵，但除超级管理员外均可改名、改代码、编辑和删除；
5. 删除使用中的角色必须选择接替角色，改派和删除保持原子性；
6. 无业务记录的普通用户可删除，有业务记录时明确提示改为停用；
7. 用户角色变更后，新权限及时生效，旧权限不能继续使用；
8. 访客只能修改本人创建的数据，直接调用 API 也不能越权操作既有数据。

#### 权限与授权

其中第 1、2 条已纳入当前自动化回归；第 3 至 6 条依赖下一阶段单次临时授权模块。

1. 前端入口显示与权限一致，但直接调用无权限 API 仍返回 403；
2. 服务员不能通过修改请求参数执行收款、退款或查看经营数据；
3. 收银员执行免单、退款、撤销收款等敏感操作时触发授权；
4. 有权限店长通过登录密码或短密码可完成单次授权；
5. 授权凭据不能跨订单、跨操作或重复使用；
6. 授权人自身无权限、已停用或会话失效时授权失败且业务数据不变。

#### 审计

1. 登录、权限变更、金额操作、导出和授权均生成审计记录；
2. 订单操作日志显示真实操作人和授权人；
3. 普通用户不能修改或删除审计日志；
4. 店长和超级管理员可按时间、用户、模块和业务对象查询；
5. 审计导出本身也必须产生审计记录。


POS 当前仍是独立应用和独立认证实现。Silver Lining OA 公共平台建立后，POS 将作为应用代码 `pos` 的首个业务系统接入统一门户、统一认证和“应用.资源.动作”权限体系。迁移必须通过兼容适配和数据迁移完成，不允许直接删除现有认证数据库或中断营业数据。

## 11. 产品决策记录

1. 邮箱服务当前不配置，后续有业务需求再启用；
2. 订单历史暂不缓存，每次进入获取最新数据；
3. 制作单为可关闭模块，关闭时全系统隐藏相关入口；
4. 优惠券售价视为实际收入，面值与售价差额视为优惠；
5. 已结清订单先撤销收款才能继续执行影响金额的操作；
6. 商品 ABC 不作为基础字段；如后续需要，应以经营分析报表计算，而不是人工维护；
7. 当前商品与分类顺序属于业务数据，任何导入或迁移必须保持用户调整后的顺序；
8. 登录同时支持账号密码和短账号 + 短密码快速登录；
9. 权限模型采用 RBAC，并内置超级管理员、店长、收银员、服务员、出品人员五类岗位角色及访客演示角色；
10. 退款、免单、撤销收款等敏感操作规划为支持店长或超级管理员单次授权（尚未实现）；
11. 权限既控制前端入口，也必须由后端 API 强制校验；
12. 用户、权限、会话和审计至少使用 SQLite，不能继续使用普通 JSON 存储；
13. 登录与权限上线后，现有订单操作日志必须记录真实操作人与授权人；
14. 除超级管理员外，初始化岗位角色和后来创建的角色一视同仁，均可编辑和删除；
15. 访客可创建演示数据，但只能操作本人创建的数据，数据归属必须由后端强制执行；
16. POS 后续接入 Silver Lining OA 统一身份平台，不在其他业务系统中复制当前认证代码。
